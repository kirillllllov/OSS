# План разработки: Веб-приложение для автоматизации ОСС

## Анализ текущего состояния

- **Фреймворк**: NestJS + Prisma + Swagger (уже настроены)
- **БД**: Prisma настроена на `sqlserver` (MS SQL Server), что соответствует требованию. В `.env` указана строка подключения к `sqlserver://localhost:1433`.
- **Готовые модули**: `Building` (полный CRUD), `Premise` (только repository), `Owner` / `Ownership` (только DTO, без контроллеров/сервисов/репозиториев).
- **Prisma Schema**: описаны только 2 модели (`building`, `premise`). Связи с будущими моделями закомментированы.
- **Swagger**: подключен, документированы базовые endpoints.
- **Авторизация**: отсутствует.
- **Workflow**: не настроен.

## Таблицы из схемы данных (sqlite.png)

Всего 21 основная таблица + связующие:

1. **Компания** (`company`) — УК, тариф, активность
2. **Сотрудник** (`employee`) — email, хеш пароля, роль, фото, 2FA
3. **Доступ_сотрудника_к_дому** (`employeeBuildingAccess`) — связь сотрудник-дом
4. **Дом** (`building`) — адрес, кадастровый номер, общая площадь (уже есть)
5. **Помещение** (`premise`) — номер, площадь, форма собственности (уже есть)
6. **Собственник** (`owner`) — ФИО, ИНН, СНИЛС, фото, описание
7. **Право_собственности** (`ownership`) — доля, площадь доли, документ-основание
8. **Контакт** (`contact`) — телефон, email, тип, уведомляемый
9. **Представитель** (`representative`) — по доверенности, дата окончания
10. **Версия_реестра** (`registryVersion`) — версия реестра собственников для дома
11. **Импорт_реестра** (`registryImport`) — загрузка файла, статус, дата импорта
12. **Собрание** (`meeting`) — форма, даты, статус, инициатор
13. **Пункт_повестки** (`agendaItem`) — вопрос, порядок, тип решения, процент принятия
14. **Библиотека_вопросов** (`questionLibrary`) — шаблоны вопросов, категории, теги
15. **Шаблон_документов** (`documentTemplate`) — тип документа, путь к файлу
16. **Бюллетень** (`ballot`) — получен/не получен, путь к скану, статус
17. **Ссылка_для_голосования** (`votingLink`) — токен, вес голоса, использована
18. **Сессия_голосования** (`votingSession`) — IP, userAgent, начало/завершение
19. **Ответ_по_вопросу** (`questionAnswer`) — ЗА/ПРОТИВ/ВОЗДЕРЖАЛСЯ
20. **Результат_кворума** (`quorumResult`) — общая площадь, участвующая, процент
21. **Сгенерированный_документ** (`generatedDocument`) — тип, версия, путь, формат

## План поэтапной реализации (с точками отката)

### Этап 0 — Финализация Prisma Schema (ВСЕ 21 модель)
**Цель**: Полная схема БД в `schema.prisma` с учетом MS SQL Server.

**Что делаем**:
- Дописать все 21 модель в `prisma/schema.prisma` с корректными связями (`@relation`, `onDelete`, `onUpdate`).
- Для MS SQL Server сохранить `@default(dbgenerated("newid()"))` для UUID.
- Раскомментировать и дополнить существующие модели (`building`, `premise`) связями с новыми таблицами.
- Выполнить `prisma migrate dev` (или `prisma db push` для dev-окружения).
- Сгенерировать Prisma Client (`prisma generate`).
- Проверить: приложение стартует без ошибок.

**Точка отката**: git commit с меткой `checkpoint: full schema`. При неудаче — `git revert` или `prisma migrate reset`.

**Критерий готовности**: `prisma generate` выполняется без ошибок, `npx prisma db pull` видит все таблицы.

### Этап 1 — Справочники базовых сущностей (CRUD)
**Цель**: Полный CRUD для всех справочников предметной области.

**Что делаем**:
1. **CompanyModule** — новый модуль (контроллер, сервис, репозиторий, DTO).
2. **BuildingModule** — доработать существующий: добавить `companyId`, связь с `Company` и `RegistryVersion`.
3. **PremiseModule** — доработать: добавить контроллер, сервис, связь с `Ownership`.
4. **OwnerModule** — доработать: добавить контроллер, сервис, репозиторий (сейчас только DTO).
5. **OwnershipModule** — доработать: добавить контроллер, сервис, репозиторий (сейчас только DTO).
6. **ContactModule** — новый модуль (контакты собственника).
7. **RepresentativeModule** — новый модуль (представители по доверенности).

**Паттерн модуля**: `controller` → `service` → `repository` → `dto` (Create/Update/Response). Все endpoints покрываются Swagger (`@ApiTags`, `@ApiOperation`, `@ApiResponse`).

**Точка отката**: каждый модуль — отдельный git commit. Можно откатить любой модуль без влияния на остальные.

**Критерий готовности**: Swagger UI (`/api/docs`) показывает endpoints для всех 7 модулей, CRUD операции работают через тестовые запросы.

### Этап 2 — Компании и сотрудники: авторизация и роли
**Цель**: Мультитенантность, JWT-аутентификация, контроль доступа.

**Что делаем**:
1. **EmployeeModule** — регистрация, вход, хеширование пароля (bcrypt), профиль.
2. **AuthModule** — JWT-аутентификация (`@nestjs/jwt`, `@nestjs/passport`), login/logout, refresh token.
3. **Guards** — `JwtAuthGuard` (проверка токена), `RolesGuard` (проверка ролей).
4. **EmployeeBuildingAccessModule** — какие дома доступны сотруднику (фильтрация по списку домов).
5. **Доработка BuildingModule** — фильтрация списка домов по `companyId` и `employeeAccess`.

**Роли** (предполагаемые): `admin` (администратор УК), `manager` (инспектор), `lawyer` (юрист), `accountant` (бухгалтер).

**Точка отката**: auth-модуль изолирован. Если JWT не работает — остальные модули могут работать без авторизации (временно). Откат — git revert на коммит auth.

**Критерий готовности**: Login endpoint возвращает JWT, защищенные endpoints требуют токен, Swagger Authorize работает.

---

### Этап 3 — Реестр версий и импорт данных
**Цель**: Управление версиями реестра собственников, импорт из CSV/Excel.

**Что делаем**:
1. **RegistryVersionModule** — создание версии реестра для дома, переключение между версиями, метка "активная".
2. **RegistryImportModule** — загрузка файла (CSV/Excel), парсинг, транзакционное создание записей `Premise` + `Owner` + `Ownership` + `Contact`.
3. **Статусы импорта**: `pending` → `processing` → `completed` / `error`.
4. **Валидация импорта**: проверка суммы долей (= 100%), проверка уникальности кадастровых номеров.

**Точка отката**: модуль импорта полностью изолирован. Откат — удаление таблиц импорта + revert commit.

**Критерий готовности**: Загрузка тестового CSV создает корректные записи в БД, сумма долей в доме = 100%.

---

### Этап 4 — Собрание (ОСС): создание и управление
**Цель**: Жизненный цикл собрания от черновика до архива.

**Что делаем**:
1. **QuestionLibraryModule** — библиотека типовых вопросов (шаблоны для повестки), категории, теги.
2. **DocumentTemplateModule** — шаблоны документов (содержимое, тип, версия).
3. **MeetingModule** — создание, редактирование, активация, завершение, архивирование собрания. Жизненный цикл: `draft` → `active` → `voting` → `counting` → `completed` → `archived`.
4. **AgendaItemModule** — пункты повестки: порядковый номер, текст для протокола/бюллетеня, тип решения (`simple_majority`, `absolute_majority`, `qualified_majority`, `unanimity`), процент для принятия.

**Точка отката**: каждый под-модуль — отдельный коммит. MeetingModule можно откатить независимо.

**Критерий готовности**: Можно создать собрание с повесткой из библиотеки, перевести в статус `active`, Swagger отображает все endpoints.

---

### Этап 5 — Голосование: бюллетени, сессии, результаты
**Цель**: Полный цикл голосования — от выдачи бюллетеня до подсчета.

**Что делаем**:
1. **BallotModule** — учет бюллетеней (получен/не получен, статус, путь к скану, ручной ввод). Связь с `Meeting` и `Premise`.
2. **VotingLinkModule** — генерация токенов для онлайн-голосования (JWT-ссылки с весом голоса), срок действия, одноразовость.
3. **VotingSessionModule** — сессии онлайн-голосования (IP, userAgent, время начала/завершения).
4. **QuestionAnswerModule** — ответы по каждому вопросу повестки (`for` / `against` / `abstain`). Связь `Ballot` + `AgendaItem`.
5. **QuorumModule** — расчет кворума в реальном времени: общая площадь дома, участвующая площадь, процент явки, решение по каждому вопросу.

**Алгоритмы расчета** (реализуются в QuorumModule):
- **Простое большинство**: > 50% от явки (при явке >= 50% от общей площади).
- **Абсолютное большинство**: > 50% от общей площади дома.
- **Квалифицированное большинство**: >= 2/3 от общей площади дома.
- **Единогласие**: 100% (отсутствие "против" и "воздержался").

**Точка отката**: модули голосования не влияют на предыдущие этапы. Можно откатить отдельно.

**Критерий готовности**: Ввод тестовых ответов корректно рассчитывает кворум и решения по всем 4 типам.

---

### Этап 6 — Генерация документов
**Цель**: Автоматическое формирование всех юридически значимых документов.

**Что делаем**:
1. **GeneratedDocumentModule** — хранение метаданных о сгенерированных документах.
2. **Генерация бюллетеней** — персональные бюллетени (ФИО, квартира, доля, вопросы) в PDF.
3. **Генерация протокола** — протокол ОСС с кворумом, явкой, решениями по вопросам, реестром проголосовавших.
4. **Реестр участников** — список собственников, принявших участие.
5. **Сообщение о проведении ОСС** — уведомление с повесткой и датами.
6. **Акт о размещении сообщения** — документ с подписями собственников.
7. **QR-коды** — на бюллетенях для онлайн-голосования (библиотека `qrcode`).
8. **ZIP-архив** — упаковка всех документов собрания.

**Точка отката**: генерация полностью независима. Использует только данные из БД. Можно откатить отдельно.

**Критерий готовности**: По завершенному тестовому собранию генерируется корректный PDF протокол и ZIP-архив с приложениями.

### Этап 7 — Инфраструктура, CI/CD, тестирование

**Что делаем**:

**Критерий готовности**: Workflow запускается, приложение доступно, seed-данные создают полноценное демо-собрание.

## Схема зависимостей этапов

```
Этап 0 (Prisma Schema)
    ↓
Этап 1 (Справочники) ←—— Этап 0
    ↓
Этап 2 (Auth) ←—— Этап 1
    ↓
Этап 3 (Реестр) ←—— Этап 1 + Этап 2
    ↓
Этап 4 (Собрание) ←—— Этап 1 + Этап 3
    ↓
Этап 5 (Голосование) ←—— Этап 4
    ↓
Этап 6 (Документы) ←—— Этап 5
    ↓
Этап 7 (Инфраструктура) ←—— Все этапы
```

## Правила отката

1. **После каждого этапа** — обязательный `git commit` с осмысленным сообщением.
2. **Перед опасными операциями** (миграция БД, удаление таблиц) — дополнительный `git tag`.
3. **При неудаче миграции** — использовать `prisma migrate reset` (в dev) или `prisma migrate resolve --rolled-back`.
4. **При конфликте схемы** — откатить Prisma schema и перегенерировать клиент.
5. **При неудаче модуля** — удалить файлы модуля и откатить commit. Остальные модули продолжают работать.

## Примечание по БД

Несмотря на то, что схема визуализирована в SQLite-формате, в проекте используется **MS SQL Server** (`provider = "sqlserver"` в Prisma). Для UUID используется `@default(dbgenerated("newid()"))`, что корректно для MS SQL Server. При необходимости деплоя на другой движок (например, PostgreSQL) — схема адаптируется заменой `dbgenerated("newid()")` на `uuid()`.
